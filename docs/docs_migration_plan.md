# docs設計書 論理的統合・移行計画書（Phase 2）

本計画書は、docs配下の設計書が肥大化・複雑化し、AIが自動生成する際に不整合を起こすリスクを排除するため、仕様の階層化（共通仕様の薄膜化、個別仕様への集約、重複排除）を行う手順と対照表（マトリクス）を定義します。

---

## 1. 移行・統合の基本方針

1. **共通仕様（Common）の薄膜化**:
   * `ui-design.md` や `css-design.md` などの共通仕様からは、具体的なレイアウト配置や個別画面の挙動、決定済みのトークン値を排除し、「サイト全体の思想・方向性」のみをシンプルに保ちます。
2. **個別画面（Pages）への集約と依存の明確化**:
   * 画面特有のCSS配置、動的挙動（吹き出し誘導、Gemini風サイドバー開閉など）は、すべて個別画面設計書（`top-page-design.md`, `category-page-design.md`, `content-page-design.md`）に集約し、各画面がどの共通仕様・トークンに依存しているかを明記します。
3. **「今後決めること」のクリーニング**:
   * すでに実装・決定済みの項目（カラーコード、ブレイクポイント、サイドバー設計など）は「今後決めること」から削除し、決定事項へ同期またはドキュメント本文に反映させます。

| 移行元ファイル (セクション) | 移行先ファイル (セクション) | 変更内容・目的 |
| :--- | :--- | :--- |

## 3. 移植漏れ検証（機械的ファクトチェック）ルール

移行作業の安全性を担保するため、以下の2段階の検証を実施します。

### ステップ 1: 一時退避による比較
変更前に `docs/archive/` ディレクトリに対象ファイルを退避し、移行作業完了後に `git diff` および `diff` コマンドで意図しない仕様の消失がないかを比較検証します。

### ステップ 2: 重要キーワードの存在チェック
退避した元ファイルに含まれていた重要キーワードや要件が、移行先ファイルに漏れなく記述されているかを `grep` 等で機械的に検証します。
* **検証対象キーワード例**:
  * `visually-hidden` (カテゴリ表示非表示用)
  * `completion-tooltip` / `読み終わったら完了を押しましょう。` (吹き出し完了誘導)
  * `var(--color-accent-muted)` / `#e6f0ef` (サイドバー背景色)
  * `is-completed` (完了カード背景色)
  * `completionId` (完了判定用ID)

---

## 4. 移行の実行ステップ

1. ユーザー様より本計画書の承認をいただく。
2. 変更対象ファイルを `docs/archive/` にコピー退避。
3. マトリクスに従い、`common/ui-design.md` などの共通仕様の薄膜化・移動を1ファイルずつ順次実行。
4. 移植漏れ検証の実施（一時退避ファイルとの比較）。
5. `repository-map.md` の責務境界とインデックスの更新・同期。
6. `docs/archive/` フォルダの削除。
7. ビルド検証（`npm run build`）およびPRの作成。
