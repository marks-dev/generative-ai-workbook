# AGENTS.md

## 厳守すべきルール

- ユーザーには常に丁寧語を使用する
- ユーザーの質問に必要な範囲でのみ説明する
- ユーザーとの対話では、不要な長文説明や過度な推測を避けること
- ただし、教材本文の作成・修正では、初心者が理解するために必要な説明量を優先すること
- 複雑な設計・実装判断・作業段取りが必要な場合は、十分に検討してから整理済みの結論のみをユーザーに伝えること
- ユーザーへの回答では、途中の思考整理をそのまま出力せず、整理済みの結論・手順・指示のみを伝えること
- ユーザーには、原則として迷わない単一の推奨手順を示すこと
- ユーザーが比較・選択肢を求めた場合のみ選択肢を提示し、その場合も最後に推奨案を明示すること
- 後付けの「こうしたらもっと良くなる」は禁止すること
- 複数ステップの作業では、開始前に作業内容と段取りを説明する
- 進捗共有が有益な場合はチェックリストを使用する
- GitHubリポジトリへの書き込み操作は、IssueのCRUD、およびユーザーが依頼した作業に必要な作業ブランチ作成・作業ブランチへのpush・Pull Request作成を許可する
- mainブランチへの直接pushは禁止する
- Pull Requestのmerge、close、review承認、release作成、tag作成、repository設定変更はユーザーの明示指示がない限り行わない
- Pull Requestを作成する場合は、変更内容、確認した設計書、実行したコマンド、確認結果、未対応事項を本文に記載する
- 作業ブランチ名は原則として `codex/` で始める
- Issue以外のGitHubリソースは、上記で許可された作業ブランチ・Pull Request操作を除き参照のみとする
- ユーザーに対してGit操作は明示的に示す
- コンテキストウィンドウを意識し、圧縮前にはAGENTS.mdルールを厳守する目的で読み直しを行うこと
- 読んでいないファイルを読んだかのように扱わないこと
- 画像生成は、明示的にユーザーから「実際に画像を生成して」と言わない限り絶対に行わないこと
- 教材の作成・修正時は、[content-format-design.md](docs/content/content-format-design.md) に定義されたガイドラインを厳格に遵守すること。

### 再現性担保と模倣禁止のルール
- **既存の教材コンテンツやコード実装を流用・参考・模倣してコードやMarkdownを寄せる（合わせる）行為を厳重に禁止します。** 既存実装と表記揺れがあろうとも、常に設計書（`docs/*.md`）の記述のみを正として新規作成・自動生成を行い、仕様に基づく再現性を担保してください。
- **既存実装ファイル（コード・Markdown）を確認する目的は、以下の2点のみに限定します。**
  1. 新規追加するコンポーネントやファイルが、既存実装と競合・衝突（デグレート）を起こさないかの確認（衝突・バグ回避目的）
  2. 既存ファイルの修正・更新時における、変更差分を最小限に抑えるための確認（差分最小化目的）

### 設計書の網羅的理解と不整合防止ルール
- 設計・実装・修正の前には、必ず `docs/repository-map.md` から対象領域の関連設計書を特定し、**関係する設計書だけでなく、docs ディレクトリ全体の設計書を一度読み込んで全体の整合性を確認してください。** 暗黙の推測や部分的な知識のみで作業を進め、設計書間の不整合や修正漏れを発生させることを禁止します。

## 設計書参照ルール

設計・実装・修正を行う場合は、以下の順に確認する。

1. AGENTS.md
2. docs/repository-map.md
3. docs/ROADMAP.md
4. 変更対象に関係するすべての設計書（※事前に一読し、仕様間の不整合がないか確認する）
5. 既存実装ファイル（※既存コードとの衝突防止、および更新時の差分確認目的のみ。模倣は厳禁）

判断に迷う場合は、docs/repository-map.md を参照して対象領域の設計書を特定する。

※設計書間で内容が重複または矛盾する場合は、[repository-map.md](docs/repository-map.md) に定義された優先順位ルールに従うこと。

AGENTS.md には設計詳細を記載せず、AIの行動ルール、設計書参照順序、実装時の判断基準のみを記載する。

## リポジトリ運用方針

- 静的ホスティングとの互換性を維持する
- 不要なビルドツールを追加しない
- 依存関係は最小限に保つ
