## 厳守すべきルール

### 2. Git・GitHub運用ルール
- ユーザーに対して実行するGit操作は必ず明示的に示す。
- 作業ブランチ名は原則として `codex/` で始める。
- mainブランチへの直接pushは禁止する。
- GitHubリポジトリへの書き込み操作は、IssueのCRUD、作業ブランチ作成・push・Pull Request作成のみを許可する。
- Pull Requestのmerge、close、review承認、release作成、tag作成、repository設定変更はユーザーの明示指示がない限り行わない。
- Pull Requestを作成する場合は、変更内容、確認した設計書、実行したコマンド、確認結果、未対応事項を本文に記載する。
- Issue以外のGitHubリソースは、上記で許可された操作を除き参照のみとする。

### 3. アセット・画像生成ルール
- 新規の画像アセット生成（generate_imageツールの使用など）は、明示的にユーザーから「実際に画像を生成して」と言われない限り絶対に行わない。なお、既存の画像ファイルのWebP変換や配置などのアセット管理操作は制限しない。

### 4. 開発・編集ルール
- コンテキストウィンドウを意識し、読んでいないファイルを読んだかのように扱わない。
- 教材の作成・修正時は、[content-format-design.md](docs/content/content-format-design.md) に定義されたガイドラインを厳格に遵守する。
- リポジトリ内のファイル（コードやドキュメントなど）に、自身の実行環境のローカル絶対パスや、APIキー、トークン、パスワードなどの機密・認証情報を書き込まないこと。

### 再現性担保と模倣禁止のルール
- **既存の教材コンテンツやコード実装を流用・参考・模倣してコードやMarkdownを寄せる（合わせる）行為を厳重に禁止します（ただし、ユーザーから明示的な指示があった場合を除く）。** 指示がない限り、既存実装と表記揺れがあろうとも、常に設計書（`docs/*.md`）の記述のみを正として新規作成・自動生成を行い、仕様に基づく再現性を担保してください。
- **既存実装ファイル（コード・Markdown）を確認する目的は、以下の2点のみに限定します。**
  1. 新規追加するコンポーネントやファイルが、既存実装と競合・衝突（デグレート）を起こさないかの確認（衝突・バグ回避目的）
  2. 既存ファイルの修正・更新時における、変更差分を最小限に抑えるための確認（差分最小化目的）

### 設計書間の不整合防止ルール
- 設計・実装・修正の前には、必ず [repository-map.md](docs/repository-map.md) から対象領域の関連設計書を特定し、変更が他の設計書と矛盾しないよう、影響範囲の設計書を読み込んで整合性を確認してください。暗黙の推測や部分的な知識のみで作業を進め、設計書間の不整合や修正漏れを発生させることを禁止します。

## 設計書参照ルール

設計・実装・修正を行う場合は、以下の順に確認する。

1. AGENTS.md
2. docs/repository-map.md
3. docs/ROADMAP.md
4. 変更対象に関係するすべての設計書（※事前に一読し、仕様間の不整合がないか確認する）
5. 既存実装ファイル（※既存コードとの衝突防止、および更新時の差分確認目的のみ。模倣は厳禁）

判断に迷う場合は、docs/repository-map.md を参照して対象領域の設計書を特定する。

※設計書間で内容が重複または矛盾する場合は、[repository-map.md](docs/repository-map.md) に定義された優先順位ルールに従うこと。

AGENTS.md には設計詳細を記載せず、AIの行動ルール、設計書参照順序、実装時の判断基準のみを記載する。

## リポジトリ運用方針

- 静的ホスティングとの互換性を維持する
- 不要なビルドツールを追加しない
- 依存関係は最小限に保つ
